## Findings
1. **Critical Dependency Block:** The plan is blocked by issue #500, which is expected to land `mesh_preflight.py`. Grep and `ls` confirm this file does NOT exist at HEAD in `src/digitalmodel/hydrodynamics/diffraction/`. The plan §"Proposed Solution" correctly identifies this but proceeds to define implementation based on "expected" API names from the #500 plan (e.g., `MESH_SURFACE_ITER`, `validate_mesh_exists`).
2. **Aggressive Collision Policy:** Plan §"Proposed Solution" item 4 states that re-running identical assets into a destination without a manifest fails as a "stale/unowned destination," even if the file content is identical to the source. This breaks common user workflows where a directory is manually prepared or a manifest is lost/deleted.
3. **Write Permission Risk:** The locking mechanism in §"Proposed Solution" item 4 uses `output_dir.parent / f".{output_dir.name}.orcawave-package.lock"`. This assumes write permission to the parent directory of the target output, which is not guaranteed in restricted environments or when `output_dir` is at a mount root.
4. **Abstraction Leak in `SpecConverter`:** Task 10 proposes refactoring `SpecConverter.convert_all()` (a generic multi-solver entry point) into an "OrcaWave-aware two-pass flow." This introduces OrcaWave-specific preflight coupling into the generic converter layer, which will scale poorly as more solvers (e.g., WAMIT, Capytaine) add strict preflights.
5. **Schema/Backend Drift for Control Surfaces:** Plan §"Scope & Impact" correctly identifies that the backend only emits mesh filenames for `VesselSpec.control_surface`, while solver flags are set via `BodySpec.control_surface`. The plan chooses to follow this drift (ignoring `BodySpec.control_surface.mesh_file` with a warning). This is a risk for #609 but a correct observation of current HEAD behavior in `orcawave_backend.py:343` and `orcawave_runner.py:503`.
6. **Atomic Rollback Limitation:** Plan §"Proposed Solution" item 2 concedes that #605 will not rollback AQWA files if a later OrcaWave staged copy/write/publish failure occurs during `convert-spec --solver all`. While it provides pre-write atomicity via preflight, mid-write atomicity is a gap.
7. **`SpecConverter.convert()` Return Type Ambiguity:** For a packaged output, `convert()` (which currently returns a file `Path`) will return the directory `Path`. The plan does not explicitly update the method's docstring or return type signature in §"Files to Change" to reflect this transition from file-path to directory-path for OrcaWave.

## Blockers
1. **Issue #500 Completion:** Implementation cannot begin until `src/digitalmodel/hydrodynamics/diffraction/mesh_preflight.py` and its associated APIs land on `main`. (Note: The plan itself includes this as a "Hard Gate" in §"Acceptance Criteria").
