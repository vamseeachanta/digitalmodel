## Findings

1. **"Cannot self-disable" is overstated and the CI cost is never weighed.** Plan lines 59–61, TDD row `test_workflow_cannot_self_disable_by_path_filter`, and the acceptance criterion "the workflow cannot self-disable" claim that removing all path filters prevents self-disable. It only guarantees the workflow *triggers*; a PR that edits the workflow can still gut its steps, and the contract test guarding workflow content runs *inside* the same workflow it guards — the same PR can delete both. Real self-disable protection is branch protection/required checks, which the plan never mentions. Meanwhile the plan buys this partial protection by running a ~150-dependency `.[dev]` install across 4 matrix legs on **every** push and PR, including docs-only commits, with no discussion of that cost or of the narrower alternative (keep filters but add `pyproject.toml`, `uv.lock`, and the workflow file to them).

2. **Acceptance criterion "Lock churn is limited to removal of cx-Oracle and hiredis" may be unsatisfiable as written.** Step 3 says "Regenerate the lock; reject unrelated broad churn," but `uv lock` re-resolution with a uv version different from the one that wrote the lock can legitimately rewrite `revision`/format or re-pin unrelated packages. The plan records no uv version anywhere (not in the repro transcript, not as a pin for regeneration), and this is explicitly a multi-machine, multi-session effort — two hosts with different uv versions can produce dueling lock rewrites. No remediation path is defined if "unrelated churn" turns out to be unavoidable.

3. **The canonical baseline command is written in a syntax that fails on the mandated host.** Verification §baseline step 3 mandates "On Windows/Python 3.12.13, run … `PYTHONPATH=src uv run python -m pytest --randomly-seed=701 --json-report …`" — a bash-style env-var prefix that is a parse error in PowerShell and does nothing useful in cmd. The plan itself uses the correct form (`$env:PYTHONPATH='src'; uv run python -m pytest`) eleven lines earlier, proving the authors know the difference; the reproducibility-critical command is the one written wrong.

4. **Fixed `--randomly-seed=701` does not deliver the determinism the subset rule assumes.** pytest-randomly's shuffle depends on the collected test set, and the final branch adds new contract tests, so baseline and final runs execute *existing* tests in different orders. `tests/conftest.py:40–42` documents that this suite already contains order-sensitive tests ("CP tests fail intermittently with random ordering (shared state issue)") — and those two are merely the known ones that got excluded. An old order-sensitive test can therefore fail in the final run but not the baseline, producing a "new/unmapped signature" that blocks closeout (Verification §baseline step 5) for a defect the PR did not introduce. The plan presents the fixed seed as making the comparison "reproducible" without acknowledging this failure mode or defining how to adjudicate it (e.g., re-run with `-p no:randomly`, or rerun-to-confirm).

5. **Round-8 is already contaminated, and the plan gives no cleanup directive.** Plan line 12 declares `scripts/review/results/issue-701-round-8/` the "current review targets" and the Files-to-Change row calls them "revision-isolated final evidence," yet the directory already contains a 0-byte `2026-07-09-plan-701-claude.md` with a retained `.err` sibling plus 305/338-byte codex/gemini stubs from a prior fanout attempt — evidence of at least one earlier run against some plan revision landing in the same round directory. The mtime-after-plan and no-`.err` promotion checks (line 345, Approval Gate) would correctly *reject* the stale claude artifact, but nothing in the plan says to delete stale round-8 contents or bump to round-9, so a round can end up holding a mix of artifacts produced against different plan revisions while still individually passing "non-empty + newer-than-plan" checks (the codex/gemini files are non-empty and may be newer than the last plan edit without corresponding to it).
