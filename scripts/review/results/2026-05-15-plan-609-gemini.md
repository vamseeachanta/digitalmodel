## Findings
1. **Model field discrepancy.** Plan §Pseudocode uses `body.id` as a component of `body_id`. `BodySpec` in `src/digitalmodel/hydrodynamics/diffraction/input_schemas.py` L632-647 does not define an `id` field; it contains a `vessel` field which has a `name`.
2. **Generic validation leakage.** Plan §Proposed Tasks 2 and 4 propose adding OrcaWave-specific validation rules (e.g., requiring `mesh_file` when `type="mesh"` for control surfaces) to `SpecConverter.validate()`. Since `SpecConverter` and its `validate` method are generic for all solvers, this could cause validation failures for specs targeting AQWA even though AQWA does not consume the `control_surface` field.
3. **Behavioral shift for unknown extensions.** Plan §Proposed Tasks 2 states that unknown control-surface mesh extensions (not `.gdf` or `.csf`) without explicit `mesh_format` will now trigger `ERROR` findings. Currently, `orcawave_backend.py` L266-269 silently defaults to `Wamit csf` for any missing or unknown format string. This will require users of other formats (e.g., `.dat`) to add explicit schema metadata.
4. **Schema field redundancy.** Plan §Proposed Tasks 7 suggests adding `panel_size` or `inner_radius` to `FreeSurfaceZoneSpec` if they are missing. `inner_radius` is already present at `src/digitalmodel/hydrodynamics/diffraction/input_schemas.py` L585; only `panel_size` is missing for the "auto" mode.
