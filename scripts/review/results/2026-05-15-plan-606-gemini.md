## Findings
1. **Unresolved dependency on missing file**: The plan's "Deliverable" and "Proposed Tasks" (Task 2, 7, 11) depend on `orcawave_asset_resolver.py` and its "package manifest writer" which do not exist at HEAD (verified via `ls` and `glob`). While acknowledged as a blocker, the specific "extension" pattern cannot be verified or tested until #605 lands.
2. **Inconsistent Naming of "Aqwa dat"**: Task 4 and Task 2 cite `Aqwa dat` as the YAML value, which matches documentation (e.g., `docs/domains/orcawave/L04_spar_benchmark/spar_benchmark.yml`). However, the plan fails to note that `orcawave_backend.py` currently maps `dat -> Wamit dat` (verified in `src/digitalmodel/hydrodynamics/diffraction/orcawave_backend.py` line 48), and other repository scripts (e.g., `scripts/benchmark/regenerate_ship_benchmark.py`) use `AQWA dat import`. The plan's choice of `Aqwa dat` is correct for YAML but requires a cleanup of these inconsistent references.
3. **Master.yml Omissions in Modular Mode**: `OrcaWaveBackend.generate_modular` currently omits `DampingLid` and `QTF` sections from its 01-08 section files; they are only merged into `master.yml` (verified in `src/digitalmodel/hydrodynamics/diffraction/orcawave_backend.py` lines 719-753). Task 10 mentions covering these in tests but does not explicitly task fixing the modular output to include them as separate files (e.g., `09_damping_lid.yml`), perpetuating an inconsistency where "modular" output is incomplete.
4. **`MeshPipeline.prepare_for_solver` Signature Conflicts**: Task 8 proposes adding `output_path` and `output_name` to `prepare_for_solver`. The current signature (verified in `src/digitalmodel/hydrodynamics/diffraction/mesh_pipeline.py` line 190) has a mandatory `output_dir`. The plan does not specify if `output_dir` becomes optional or how it interacts with an explicit `output_path`, leading to potential ambiguity in the new API contract.
5. **FDF Validator Implementation Gap**: Task 6 proposes using `_parse_fdf_panels` as a "row parser". However, `_parse_fdf_panels` (verified in `src/digitalmodel/hydrodynamics/diffraction/benchmark_helpers.py` line 55) is a whole-file parser that returns a nested list and skips the first 4 lines. It cannot be used as a row-level primitive without refactoring `benchmark_helpers.py`, which is not listed in the "Files to Change" table for that reason.
6. **Backend Blessing of Unsupported Formats**: The plan (Task 3) states `.obj` and `.msh` are "fail-closed". However, `_MESH_FORMAT_MAP` in `orcawave_backend.py` (line 49-50) explicitly maps them to `"Gmsh"` and `"OBJ"`. Unless these mappings are removed or the backend is strictly forced to use the `prepared_metadata_map` for all assets (including those that might bypass the preparer), the backend will still produce YAML for these "unsupported" formats.
