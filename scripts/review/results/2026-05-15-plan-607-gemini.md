## Findings
1. **Minor Inconsistency in CLI Conventions:** Plan §Deliverable and §Proposed Task 2 define `--output/-o` for `mesh-to-spec` as a file path (defaulting to `<mesh.stem>.diffraction.yml`). This deviates from the repository convention seen in `convert-aqwa`, `convert-orcawave`, `compare`, `convert-spec`, and `mesh-build` where `-o` always denotes an output directory. While acceptable for a single-file generator, it may surprise users accustomed to the directory-based `-o` pattern.
2. **Conservative Mesh Format Support:** Plan §Proposed Task 4 and §TDD Test List (last item) intentionally "fail closed" for `.obj` and `.msh` formats despite them being defined in `input_schemas.py:MeshFormatType` and mapped in `orcawave_backend.py:_MESH_FORMAT_MAP`. This is a self-imposed constraint to align with issue #606, but it technically restricts existing backend capabilities.
3. **Limited Inertia Configuration:** Plan §Proposed Task 2 and §Proposed Task 5 focus on `mass`, `cog`, `rog`, and `inertia_tensor` but omit CLI flags for `VesselInertia.mode` or `VesselInertia.inertia_tensor_origin`. Users requiring `free_floating` mode or `centre_of_mass` origin must use a template or rely on the defaults (`explicit` and `body_origin`). This is appropriate for a T2 plan but should be documented in the help text.
4. **Stale Documentation Context:** Plan §Proposed Task 10 correctly identifies that `docs/domains/orcawave/README.md` contains stale workflow links and broken references (e.g., `diffraction/QUICK_START.md` which does not exist at HEAD). The plan's decision to defer or consume #614's cleanup is a necessary dependency for high-quality documentation.
