# Re-review artifact: digitalmodel #596 revised repo-structure plan — reviewer 3

Verdict: MAJOR

Findings:
- [severity: high] No mandatory CI enforcement path for the new repo-structure checker/pre-commit gate — The plan adds a local hook and a direct command, but it never binds either to a named GitHub Actions workflow. Live repo evidence shows no existing workflow runs `pre-commit`, `verify_repo_structure.py`, or `verify-repo-structure`, so this could remain a local-only guard and be bypassed in PRs/merges. Required change: add an explicit workflow row and hard CI command in a named workflow (for example `quality-gates.yml` or a new dedicated workflow) that runs the checker or `pre-commit` on PR/push and fails the build on violations.
- [severity: high] Docs gate is weaker than the plan claims for this specific migration — `docs.yml` currently triggers only on `src/**`, `docs/api/**`, and `mkdocs.yml`, but this plan will touch `docs/standards/**`, `docs/examples/**`, and likely `docs/domains/**`, so the repo’s docs CI would not automatically run for the approved docs moves. Separately, `mkdocs build --strict` is not enough to prove the moved B1528 links are valid, and the plan only requires `pre-commit run verify-repo-structure --all-files`, not the existing `markdown-link-check` or full pre-commit suite. Required change: expand docs CI path filters to include all touched docs surfaces for this issue and add a hard docs-link gate such as `uv run pre-commit run --all-files` or at minimum `markdown-link-check` plus the routing tests after link rewrites.
- [severity: medium] Approval marker/governance is materially better, but still lacks an executor preflight that binds implementation to the exact approved plan revision — The plan now specifies marker contents and correctly states that a local marker alone never authorizes implementation, which is a strong improvement. However, it still does not require a pre-implementation verification that the current plan file SHA matches the SHA recorded in `.planning/plan-approved/596.md`, so a post-approval plan edit could drift from what was actually approved. Required change: add an explicit preflight gate before any implementation commit that compares the approved plan SHA/path in the marker against the current plan blob/commit SHA and aborts on mismatch.
