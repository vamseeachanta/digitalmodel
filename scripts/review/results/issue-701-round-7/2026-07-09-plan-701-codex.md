## Findings
1. **MAJOR — The workflow contract can disable its own enforcement.** Plan §Workflow contract lines 311–320, Implementation Step 5 lines 386–389, and Acceptance Criterion lines 507–508 claim workflow-only changes execute the dependency contract. Both the trigger and the contract invocation reside in `.github/workflows/workflow-automation-tests.yml`; a workflow-only change can remove its own path entry and contract step, causing the workflow to be skipped. No independent guard runs the new file: `.github/workflows/quality-gates-by-domain.yml` runs only three existing harness tests, and `scripts/ci/detect_touched_domains.py:51-60` does not map this workflow to the contracts domain. GitHub evaluates changed files against configured path filters, while `pull_request` workflows execute from the PR merge context. [GitHub path-filter documentation](https://docs.github.com/en/actions/how-tos/write-workflows/choose-when-workflows-run/trigger-a-workflow), [GitHub pull-request event documentation](https://docs.github.com/en/actions/reference/workflows-and-actions/events-that-trigger-workflows). Put the contract in an independent always-running harness or remove the self-selecting path filter.

2. **MAJOR — The metadata contract omits dependency surfaces that can reintroduce the forbidden packages.** Plan §Dependency contract lines 288–299 scans only `[project].dependencies` and `[project.optional-dependencies]`. `pyproject.toml:402-404` also has `[dependency-groups].dev`, while `pyproject.toml:1-2` has build requirements. The frozen workflow install, `uv pip install --no-sources -e ".[dev]"`, installs the optional extra, not the PEP 735 dependency group, and the workflow does not run `uv lock --check`. Consequently, adding `oracledb` or `hiredis` to the dev group while leaving `uv.lock` stale passes every proposed target-workflow assertion. Scan all dependency-bearing tables, including nested dependency-group includes, and enforce lock freshness in CI.

3. **MAJOR — The full-suite comparison still has no reproducible baseline.** Plan §Verification lines 449–454 and Acceptance Criterion lines 509–510 refer to a “current-main baseline tracked by #700.” `docs/plans/2026-06-11-issue-700-ci-baseline-domain-gates.md:28-30` records only failing shard names and workflow run IDs—not the canonical command’s nodeids, signatures, environment, or report artifact. Current main cannot produce that Windows baseline with the canonical command because the plan’s own reproduction shows sync failing before pytest. `pyproject.toml:96` also enables `pytest-randomly` without a fixed seed. Define an exact main SHA, OS/Python, fixed seed, machine-readable report, signature normalization, and a same-environment post-change comparison; otherwise “no new failures” remains subjective.

4. **MINOR — The hiredis runtime rationale is false for the locked graph.** Plan Resource Intelligence lines 33–38 claims hiredis is “indirectly consumed” and removal may reduce parser throughput. `uv.lock` resolves `redis==7.2.0` with `hiredis==2.2.3`; empirical import returned `redis.utils.HIREDIS_AVAILABLE=False` because Redis 7.2 requires hiredis 3.2 or newer. The current pin is installed but not selected, so the stated throughput risk and performance-reintroduction rationale are unsupported.

5. **MINOR — The mandatory central-index change has no landing path.** Plan §Files to Change includes `../workspace-hub/docs/plans/README.md`, but that row is currently an uncommitted modification in a separate Git repository. Implementation Step 8 only pushes the digitalmodel branch and opens its PR, which cannot contain the sibling-repository change. Specify a separate workspace-hub commit/push and ownership sequence.

6. **MINOR — The disposable “clean” test run leaks a sibling source checkout.** Plan Verification lines 472–476 runs pytest from this repository, while `tests/conftest.py:51-53` inserts `../assetutilities/src` ahead of the environment installed with `--no-sources`. On the evidence host, workflow tests therefore use the sibling assetutilities source rather than the publishable dependency resolution the clean install is meant to validate. Run this proof in a clone without the sibling override or explicitly verify the imported assetutilities path.
