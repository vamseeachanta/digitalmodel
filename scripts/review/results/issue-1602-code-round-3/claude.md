### Verdict: MAJOR

### Summary
Third-round adversarial review of the #1602 Task 1 validator, with counterexamples executed against the real contract artifacts in the worktree. The prior MAJOR classes (owner/consumer/closeout/DAG/commitment/duplicate-path, envelope-extension overwrite, TOCTOU read, symlink/output aliasing, atomic emit) are genuinely closed: all four parsed input documents are content-pinned by independent fingerprints checked last, so no semantic output-producing bypass was found. Two demonstrated MINOR-grade gaps remain. (1) Presentation bypass on the root manifest itself: reordering mapping keys inside effective_authority in issue-1602-riser-analysis-contract-v3.yaml passes all validation (fingerprints use order-insensitive json.dumps(sort_keys=True)) yet emits a byte-different composed.yaml (verified: outputs differ at the authority block), because _atomic_emit dumps with sort_keys=False and composed key order flows from manifest bytes. Component presentation is pinned via raw SHA commitments in the fingerprinted root, but the root's own bytes are hash-committed nowhere — so two byte-distinct "approved" composed artifacts can exist under the same approved identity, undermining the stated fresh-artifact-identity property (semantics unchanged; parsed content is still exact). (2) Untyped-crash class: validate_contract/validate_references run outside the KeyError/TypeError/ValueError→ContractValidationError wrapper that protects only _compose; deleting milestone_DAG_extensions.deterministic_bundle_readiness_guards (host) or effective_public_dataset_surfaces (root) crashes with raw KeyError tracebacks (verified). Both crashes are fail-closed — they occur before emission and no output is produced — but they violate the typed-error contract that test_unknown_child_extension_returns_typed_error asserts. Fixes are small and localized. VERDICT: MINOR.

### Issues Found
- CONFIRMED (executed): root-manifest presentation change produces a byte-different approved output. Reordering keys inside effective_authority in the root manifest passes all validators — including validate_approved_fingerprints (src/digitalmodel/contract_validation/riser_exact_semantics.py:191-193 hashes json.dumps(sort_keys=True), which is key-order-insensitive) — and validate_parent_contract emits composed.yaml whose bytes differ from the baseline run (first divergence in the authority block). Cause chain: _compose copies root["effective_authority"] into composed (riser_parent_contract.py:208) preserving manifest insertion order, and _atomic_emit dumps with sort_keys=False (riser_parent_contract.py:355). Component files are presentation-pinned via their raw sha256 commitments inside the fingerprinted root, but the root manifest's own bytes are never hash-committed, so 'presentation changes require fresh approved artifact identity' does not hold for the root itself. Impact is byte-identity only (parsed semantics remain exact), but any downstream commitment to the composed file's sha256 can now bind two different byte artifacts to one approval.
- CONFIRMED (executed): untyped KeyError crashes on structurally malformed input. The KeyError/TypeError/ValueError→ContractValidationError wrapper in validate_parent_contract covers only _compose (riser_parent_contract.py:388-393); validate_contract at line 394 and validate_references run unwrapped. Verified crashes: deleting milestone_DAG_extensions["deterministic_bundle_readiness_guards"] from the host component → raw KeyError in _validate_dag_guards (riser_reference_rules.py:123-126); deleting effective_public_dataset_surfaces from the root → raw KeyError in _validate_required_surface (riser_contract_rules.py:80). Both are fail-closed (no output emitted, exception precedes _atomic_emit) but escape the CLI's typed-error handling in scripts/validation/validate_riser_parent_contract.py:31-34, producing tracebacks instead of the 'contract validation failed' exit path, contradicting the intent shown by test_unknown_child_extension_returns_typed_error (tests/validation/test_riser_parent_contract_semantics.py:2455-2465 in diff numbering).

### Suggestions
- Close the root-presentation gap by making the emitted artifact canonical rather than input-order-dependent: dump with yaml.safe_dump(composed, sort_keys=True) in _atomic_emit (riser_parent_contract.py:355), or deep-normalize composed key order before emission; alternatively (or additionally) commit the root manifest's raw sha256 out-of-band (e.g., in the approval record) the same way component bytes are committed.
- Extend the structural-error wrapper to cover validation as well as composition: move validate_contract(root, base, components, composed) (riser_parent_contract.py:394) inside the existing try/except so KeyError/TypeError/ValueError surface as ContractValidationError('invalid contract structure: ...'), keeping the CLI's typed fail-closed exit path.
- Harden _record_hash (riser_exact_semantics.py:191-193) against JSON's silent key coercion: json.dumps coerces int/float/bool/None mapping keys to their string forms, so {1: x} and {"1": x} (or {True: x} and {"true": x}) fingerprint identically, and mixed-type keys raise an uncaught TypeError under sort_keys=True. Reject non-string mapping keys at YAML load time in _UniqueKeySafeLoader — this eliminates both the theoretical collision class and the crash class in one check.
- Windows portability (repo runs on D:\workspace-hub): (a) resolve_bound_path can hit os.path.commonpath ValueError (uncaught, untyped) for drive-relative declared paths like 'C:foo', which pass the is_absolute() guard on Windows (riser_manifest_rules.py:47-51); (b) the post-replace directory-fsync open in _atomic_emit (riser_parent_contract.py:363-367) fails with OSError on Windows after os.replace has already succeeded, raising 'cannot atomically emit output' even though the validated output was written — guard the directory fsync with a POSIX check or tolerate its failure.

### Questions for Author
- Is byte-level identity of the composed artifact (e.g., a downstream sha256 commitment over composed.yaml) part of the approved-identity model, or is only parsed-content identity in scope? test_cli_output_is_stable_across_hash_seeds suggests byte determinism matters, in which case the root key-order dependence is in scope for the fingerprint guarantee.
- Is a raw traceback (vs the typed 'contract validation failed' CLI exit) acceptable for structurally malformed inputs given the crash is provably pre-emission, or should the typed-error contract asserted in test_unknown_child_extension_returns_typed_error hold for all malformed-input paths?
- Is Windows an execution target for this validator (the repo memory indicates D:\workspace-hub usage)? If so, the commonpath ValueError and post-replace directory-fsync failure paths deserve regression tests.
