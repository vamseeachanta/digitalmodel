## Findings
1. **Task 5 & 12: Mixing Pydantic and Dataclasses.** Task 5 considers reusing `LoadRAOData` (a Pydantic model from `report_data_models.py`) inside `DiffractionResults` (a dataclass from `output_schemas.py`). This may lead to inconsistent serialization patterns (`model_dump()` vs `to_dict()`) and requires the proposed `diffraction_results_io.py` loader to handle these hybrid types explicitly when reconstructing NumPy arrays.
2. **Task 9: Heading unwrap policy constraints.** The plan forbids seam interpolation (crossing 0/360) unless `fixture_manifest.yml` sets `heading_unwrap: true`. While a strong safety measure for verification fixtures, the plan does not explicitly add a corresponding setting to the user-facing convergence-study YAML schema, which may result in unexpected `UNAVAILABLE` metrics in production studies involving periodic grids.
3. **Task 5 & 8: Exception Class Definitions.** The plan references specialized exceptions `OrcaWaveConvergenceDependencyError` and `OrcaWaveConvergenceDataError` but does not list a file for their definition. They should be explicitly added to the new `orcawave_convergence.py` or a shared `exceptions.py` to avoid circular imports.
4. **Task 3: Batch YAML compatibility mechanism.** The requirement for a "raw YAML key whitelist check" is a high-signal defensive measure. It prevents the dry-run planner from emitting batch configurations that contain "ghost" fields ignored by Pydantic but invalid for the actual `OrcaWaveBatchRunner` execution.
5. **Task 10: Contextual Memory Estimates.** The decision to treat `run_time.md` constants as "reference observations" rather than universal limits correctly addresses the risk of OrcaWave performance variability across versions and hardware, favoring operator-supplied calibration.
