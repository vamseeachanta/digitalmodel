`★ Insight ─────────────────────────────────────`
- A repo-structure plan must lock its own evidence claims before approval — when `if existing; otherwise create` language appears against a path that *does* already exist, the planner has not actually finished resource intelligence and the gate is procedurally weak.
- The `.gitignore`-vs-`git ls-files` cross-check is the highest-value adversarial check on this class of plan: any path that is both ignored AND tracked is a pre-existing inconsistency that will silently mis-route a migration command.
- "Defer to implementation" is acceptable for tactics (e.g. exact diff of a function body); it is not acceptable for structural choices like checker location, contract document path, or allowlist contents — those are the *plan*.
`─────────────────────────────────────────────────`

I spot-checked the plan against the live `digitalmodel` tree on `main`. Several evidence claims do not match repo state, and the migration matrix has internal contradictions that the planner did not flag.

# Adversarial Review — digitalmodel #596

**Verdict: MAJOR**

## Findings

### Critical

- **[critical] `outputs/` is tracked AND `.gitignore`-listed simultaneously — plan never notes the contradiction.** `.gitignore:358` contains `outputs/`, yet `git ls-files outputs/` returns 18 tracked files under `outputs/b1528_sirocco/` (force-added at some point). The migration matrix says "Classify as durable fixture vs generated output before moving" without flagging that the path is already in an inconsistent state. **Required change:** add an explicit "pre-existing inconsistency" finding to Resource Intelligence; lock a binary decision in the plan — either (a) remove `outputs/` from `.gitignore` and reclassify `outputs/b1528_sirocco/**` as a durable fixture/report directory with a name that actually reads as durable, OR (b) `git rm --cached outputs/b1528_sirocco/**` and document a regeneration command + bulk-store target. Do not enter implementation with this still as "Classify... before moving."

- **[critical] Plan's domain-asset deferral is built on a misclassification.** Resource Intelligence lists `reports/`, `memory/`, `specs/`, `benchmarks/`, `notebooks/`, `projects/` under "Tracked top-level distribution" via the `EXISTS` check, and the migration matrix calls them "domain asset assets requiring classification … may carry domain knowledge." Live state: `git ls-files reports/`, `git ls-files memory/`, `git ls-files specs/` all return **zero tracked files**. They exist only as untracked local roots. The "do-not-move-in-first-pass" risk argument therefore evaporates — there is nothing to move; classification is trivial (`.gitignore` confirmation or removal), not a follow-up risk. Worse, `reports/` is referenced by CI (`reports/quality_gates_results.json`, `reports/quality-gates-pytest-full.log`) so it is a *generated-output landing zone for CI*, not a "domain asset" — different semantics, different action. **Required change:** re-run the inventory using `git ls-files <dir>/`, separate tracked-domain-content dirs from untracked-runtime-output dirs, and re-classify `reports/` explicitly as a CI artifact-landing path.

### High

- **[high] Resource Intel is wrong about `tests/docs/test_digitalmodel_routing_contract.py`.** Plan says "extend `tests/docs/test_digitalmodel_routing_contract.py` if existing; otherwise create targeted tests." The file **already exists** (`tests/docs/test_digitalmodel_routing_contract.py`, 4226 bytes, last modified 2026-05-03). A plan-stage gate that says "if existing" has not actually checked. **Required change:** assert existence in Resource Intelligence with an `ls`/`git ls-files` line, and lock the action to "extend," not "extend or create."

- **[high] Root-utility scope is incomplete.** Plan covers `_count_tests.py` and `_coverage_map.py` but `git ls-files` of root tracks **two more** Python utilities at root: `orcaflex_maturity_analysis.py` and `vulture_whitelist.py`. Both are stylistically root-utility scripts and would naturally be governed by the same allowlist rule. The plan must either include them in the migration matrix with an action, or explicitly state why they are excluded (e.g. `vulture_whitelist.py` is conventional vulture config and stays at root). **Required change:** enumerate every tracked root `*.py` and assign each one an action.

- **[high] `config/repo_structure.yml` content is undefined.** The plan says it will exist and tests reference it, but no draft schema or allowlist content is in the plan. Adversarial reviewers cannot assess whether the contract correctly classifies `outputs/`, `reports/`, AI-agent surfaces, etc. Without seeing the rules, "the structure checker passes" tells us nothing. **Required change:** include a draft YAML in the plan body — at minimum the root-allowlist set, the generated-pattern denylist, and the durable-fixture allowlist exceptions — so the reviewer is approving rules, not aspirations.

- **[high] Structural decisions are deferred to "during implementation."** The plan explicitly leaves *four* path decisions open: (1) `docs/standards/repo-structure.md` vs a `docs/README.md` section, (2) `scripts/verify-repo-structure.py` vs `scripts/maintenance/verify_repo_structure.py`, (3) `config/repo_structure.yml` vs `config/quality/repo_structure.yml`, (4) plan-index policy. These are structural choices, not implementation tactics. A `status:plan-review` plan should ship with each one resolved and justified. **Required change:** pick each path now with one-line rationale tied to existing repo organization, and remove the "or" alternatives.

- **[high] No invocation surface for the new checker.** Plan creates a verifier but never says where it runs: pre-commit hook? CI job in `.github/workflows/`? Manual `make` target? Local-only? Without an invocation surface, the structure contract is a passive document, not a guardrail — and any drift will recur immediately. **Required change:** decide at least one binding surface (pre-commit recommended for tier-1; CI job acceptable as floor) and add the wiring as a Files-to-Change entry.

### Medium

- **[medium] CI workflow modification trigger conditions are not enumerated.** Plan says modify `.github/workflows/*.yml` "only if generated-report locations change" but does not list which generated paths are in scope (`coverage.xml`, `coverage.json`, `dist/`, `reports/quality_gates_results.json`, `reports/quality-gates-pytest-full.log`) nor specify a verification step that CI uploads still resolve after path changes. **Required change:** enumerate the CI-referenced paths in the matrix, mark each as "no-op" or "modify with diff preview," and add a CI-dryrun verification step.

- **[medium] Reference-search method is unspecified.** "Reference search proves no stale references" appears multiple times with no command defined. `git grep`, `rg`, cross-repo workspace-hub references? Hits inside `docs/`? Inside YAML config? **Required change:** specify the exact reference-search recipe (e.g. `rg -F '<old_path>' --hidden -g '!.git'` plus `rg` over workspace-hub) and bound the search scope.

- **[medium] `coverage.xml` / `coverage.json` policy is ambiguous.** They are CI-referenced upload paths, generated, not currently tracked, not in the migration matrix. Plan does not say whether they should be explicitly ignored, routed under `reports/`, or left alone. **Required change:** add a row for both with an action.

- **[medium] `mkdocs build --strict` is conditional.** "If environment supports it" undermines verification. Any plan that touches docs surfaces (and this one creates `docs/standards/repo-structure.md`) must require docs build, not make it optional. **Required change:** make `uv run --group docs mkdocs build --strict` a hard verification gate; if the local env can't run it, gate on the CI `docs.yml` workflow result instead.

- **[medium] No rollback/recovery procedure.** A path move that breaks CI or docs needs an explicit revert recipe, particularly because the plan describes a single "transactional closeout window." **Required change:** add a Rollback section: per-action `git revert` target, per-CI-edit fast-revert path, and what proof is required to declare rollback successful.

- **[medium] Branch/PR strategy not stated.** Tier-1 structural change touching CI, docs, root files, and tests deserves explicit branching: single PR or per-action splits, reviewer set, and squash policy. **Required change:** add a one-paragraph "Branch and Commit Strategy" section. Default recommendation: feature branch `chore/596-repo-structure`, atomic commit per migration-matrix row with the row name in the subject, single PR.

- **[medium] Approval marker tracking semantics not stated.** `.planning/plan-approved/*.md` files **are tracked** (83 entries already in git). Plan only says marker "exists." Reviewers (and ecosystem reuse later) need to know the marker is durable and merged, not local-only. **Required change:** state that `.planning/plan-approved/596.md` will be committed.

### Low

- **[low] TDD list lacks negative-path tests.** No test that the checker emits a clear, specific error message when a violation is found; no test that `module-routing.yaml` parses cleanly. **Required change:** add a "checker error message contains path and rule name" test and a YAML-load smoke test.

- **[low] Ecosystem-export hook missing.** Parent anchors are workspace-hub#1962 and #2397 — this is the *first* repo-by-repo refactor. Plan does not say what gets exported back to workspace-hub as a reusable template (allowlist schema? checker pattern? migration matrix template?). **Required change:** add a "Workspace-hub upstream artifacts" subsection naming the specific files that feed the next repo's plan.

- **[low] Atomic-commit policy unstated.** "Closeout window" implies one big commit; bisectability for a structural refactor benefits from per-row atomic commits. See the Branch/PR strategy fix above.
