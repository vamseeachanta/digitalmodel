Diffraction Domain Architecture

Output-driven analysis contract · OrcaWave (pilot) + AQWA · digitalmodel

Date: 2026-05-27 Status: design / proposal Scope: src/digitalmodel/hydrodynamics/diffraction/ Companion: output-driven reframe of DIFFRACTION_CAPABILITIES_EXPANSION_PLAN.md
Domain objective (all analysis domains, all analysis repos). Given (1) a defined output/outcome and (2) raw data — which may be barebones or fully detailed — the domain module shall:
  • a Q&A + assume missing data — detect what the requested outcome needs, ask the user, and where unanswered supply suitable assumed values drawn from maintained reference data.
  • b prepare input data — assemble the solver-ready input package from the now-complete dataset.
  • c provide output with sanity checks — run, then return results gated by physical / range / coverage validation.
This is an output-driven (inverse) contract — outcome defined first, inputs inferred — not the usual input-driven spec → run → results flow.
★ The load-bearing requirement. The module must work whether the user supplies almost nothing or a complete dataset. The non-negotiable is that every value the module assumed (rather than received) is highlighted in the output report and brought to the user's attention, with its source and engineering basis. The report is the place where assumptions become visible and reviewable — silent assumptions are forbidden.

1 · The data spectrum

The same entry point accepts any point on this spectrum; the assumption engine fills the delta.

4 · Assumption ledger — the heart of pillar (a)

Every resolved field carries provenance. This is what makes assumptions visible in the report.

AssumptionRecord:
  field:        "bodies[0].inertia.radii_of_gyration"
  value:        [12.4, 31.8, 33.1]   # m
  source:       user_supplied | assumed_default | estimated_from_data | database_lookup
  basis:        "estimate_radii_of_gyration(profile) — empirical k=0.34·B, 0.30·L"
  reference:    "hull_catalog: barge-class / parametric_spec_generator.py:173"
  confidence:   high | medium | low
Reconciliation with #525. Issue #525 removed silent defaults from the OrcaWave parser (it now raises on missing data). That is fully compatible with this contract: assumed values are permitted only when recorded in the ledger and surfaced. "No silent defaults" and "fill missing data" are the same rule viewed from two sides — the ledger is the bridge.

The output report gains a mandatory Assumptions section listing every record where source ≠ user_supplied, sorted by impact/confidence, so the engineer reviews exactly what the module decided on their behalf before trusting the results.

5 · Maintained reference data feeds pillar (a)

Per direction: assumption quality improves over time as example/analysis databases are maintained and grow.

  • Hull library (hydrodynamics/hull_library/) — catalog + parametric profiles → mass/CoG/gyradii lookups.
  • RAO database (rao_database.py) — prior-analysis RAOs indexed by hull parameters → first-guess responses & cross-checks.
  • Example sets (docs/domains/orcawave/L00–L06, AQWA benchmark) — canonical worked cases → templates for grids & solver options.
  • WAMIT / analytical reference (wamit_reference_loader.py, L00_validation_wamit) → validation baselines for pillar (c).

The resolver consults these in priority order (exact match → parametric estimate → conservative default), tagging each result's source/confidence accordingly.
