Goal and boundary

The implementation will consume exactly one provenance-complete public MODU source record from worldenergydata #1050 and will emit a deterministic, solver-neutral FloatingHostIdentityEnvelope that resolves one physical vessel, hull geometry, and loading condition. It will reference the WED source and digitalmodel hull records; it will not create another fleet master, execute diffraction, or infer identity from a rig name.

Execution gates: implementation will remain blocked until WED #1050 lands an accepted source contract, digitalmodel PR #1627 resolves its open fingerprint review defect and lands the parent validator, this plan receives no-MAJOR adversarial review, and the user explicitly applies status:plan-approved.

Resource intelligence summary

  • src/digitalmodel/marine_ops/vessel_db/wed_adapter.py adapts legacy fleet CSV rows into name-oriented Record objects and does not preserve this contract's complete commitments. #1618 will not consume that lossy adapter: WED #1050 will own and emit a separate typed PublicFloaterSourceRecord artifact/API, which this issue will consume directly.
  • src/digitalmodel/marine_ops/vessel_db/loader.py normalizes vessel particulars and aliases, while mass_properties.py, gyradii.py, and the hull-library catalog provide reusable evidence inputs. None is identity authority.
  • The packaged WED drilling-rig CSV currently contains 2,268 rows, zero populated IMO values, 1,673 rows with LOA and beam, and 1,540 with LOA, beam, draft, and displacement. It contains no CoG or inertia columns. These rows cannot satisfy this envelope until #1050 supplies the required provenance and mass/loading evidence.
  • Record.vessel_id() and existing hull catalog keys are name/file-derived aliases. mass_properties.from_record() may estimate gyradii and default missing CoG coordinates; neither behavior may establish physical identity or actual/public eligibility.
  • docs/plans/issue-1602-riser-host-motion-contract-v3.yaml fixes the exact minimum bindings and producer boundary for floating_host_identity.
  • WED #1050 is open at status:needs-plan; its output will therefore be a hard precondition, not a fabricated fixture promoted as public evidence.
  • The drive-index query public MODU drilling rig vessel hull identity loading condition IMO ran on 2026-07-18 through all six configured indexes. It found historical/private drilling-rig and vessel references, but no result may seed a public host row. Several indexes were stale; absence claims will not be inferred from that search.
  • Standards: identity resolution is a provenance/data-contract task. No standards-derived calculation constant will be introduced; the calc citation contract is not triggered.
  • Wiki: no wiki content will be touched. Reproduction: N/A — this issue will add a new envelope rather than repair an alleged runtime regression.

The semantic commitment preimage will contain every parent minimum binding except floating_host_identity_envelope_sha256. YAML will first decode through a strict duplicate-key-safe loader into the exact typed child model; implicit timestamps, custom tags, booleans-as-numbers, NaN/infinity, and schema coercions will be rejected. The normalized tree will permit only NFC UTF-8 strings, booleans, null, arrays, mappings with string keys, and integers where the schema explicitly declares an integer; all engineering real values will be canonical decimal strings. RFC 8785 will then define key ordering, escaping, and UTF-8 JSON serialization. A frozen golden preimage/byte/hash vector will be independently reconstructed in tests. Only transport-file and YAML-presentation hashes outside the parent interface will be excluded; every parent-required semantic evidence/content commitment, including the source commitment and hull/loading/evidence hashes, will remain in the envelope preimage.

The canonical host body frame will be a local right-handed SI frame: +x forward, +y port, +z upward, with an explicit evidenced physical origin. A hash-bound rigid transform will map this local frame to the parent-owned global ENU_right_handed frame. Source/catalog-to-local and local-to-ENU transforms, signs, origins, and units will be independently tested; no implicit solver origin will be accepted.

ReviewedHostResolutionEvidence will load from the independent immutable resolution-selection artifact, never from the crosswalk/envelope that the validator emits. #1618 will own that selection artifact. An independent engineering reviewer will approve or reject its exact predicates through an immutable review receipt before the validator may emit an eligible crosswalk/envelope. The selection artifact will bind exact equality to the producer's rig_source_commitment_sha256, physical_identity_evidence_sha256, loading_condition_source_evidence_sha256, host_particulars_source_evidence_sha256, and mass_properties_source_evidence_sha256, plus the chosen physical-vessel ID/evidence, hull artifact ID/geometry hash, and loading ID/hash. verify_host_resolution_evidence will verify the receipt and perform an exact hull ID/geometry-hash lookup against HullCatalog; loading equality will be verified against producer evidence rather than a panel/mesh catalog. Only its VerifiedHostResolution output may enter resolve_floating_host_identity. Automatic name, dimensional-similarity, generic HULL_LIBRARY_REF, or heuristic matching will be forbidden for promotion and may produce only a non-promotable disposition.

The review receipt will bind selection_sha256 (RFC-8785 hash of every selection field), reviewer authority ID/role, APPROVE or REJECT disposition, reviewed Git commit SHA, digitalmodel issue number, GitHub comment database ID, comment-body SHA-256, comment updated_at, and receipt schema version. review_receipt_commitment_sha256 will cover every receipt field except itself. The CLI boundary will call fetch_and_verify_remote_review through an injected GitHubReviewClient, require an allowlisted repository-owner/reviewer identity and exact issue/comment/body/selection/disposition/timestamp matches, and produce a typed VerifiedRemoteReview. The pure deterministic verify_host_resolution_evidence function will receive the selection, separate receipt, explicit verified-remote value, and hull catalog; it will perform no file or network I/O. Fetch time and transport metadata will remain non-canonical execution evidence. Offline or unverifiable remote state will be non-promotable. Mutation of any selection, receipt, or verified-remote field after review will fail before envelope emission; no private signing key will enter the repository.

TDD implementation sequence

  1. RED — exact schema and commitments. Tests will require every v3 minimum binding, reject unknown/duplicate semantic fields, reject a source commitment mismatch, and prove that key order/YAML formatting cannot change the semantic envelope commitment. Command: uv run pytest tests/marine_ops/vessel_db/test_host_identity.py -x -v; expected initial result: failures because the module will not exist.
  2. GREEN — minimal immutable models. The implementation will add typed immutable records, strict typed-to-canonical JSON-compatible normalization, explicit units/frame records, and exact commitment verification. The focused tests will then pass.
  3. RED — identity resolution. Fixtures will cover matching IMO, reviewed fallback identity, name-only collision, conflicting IMO, alias-to-different-unit collision, zero/multiple candidate matches, and loading-condition mismatch.
  4. GREEN — fail-closed crosswalk. Resolution will require an independently reviewed selection artifact for exactly one physical unit, one immutable hull artifact, and one loading condition; every rejection will return an explicit typed disposition without emitting an eligible envelope.
  5. Integration. The host-identity test will prove the accepted #1050 record commitment is preserved byte-for-byte through the resolver input, reviewed catalog evidence resolves through exact immutable hashes, child YAML/model/CLI/preimage declarations are bidirectionally identical, and emitted binding names equal the parent v3 contract.

Risks and review focus

  • Upstream drift: #1050 may land a field shape different from this plan. Implementation will consume its reviewed contract and will revise this plan if that changes the parent interface.
  • Identity false-positive: IMO reuse/data errors and aliases are not sufficient. Review will attack collision and fallback-evidence rules.
  • Fingerprint coupling: PR #1627 currently has an open review finding on raw hash fields contaminating semantic fingerprints. This plan will depend on the corrected parent primitive.
