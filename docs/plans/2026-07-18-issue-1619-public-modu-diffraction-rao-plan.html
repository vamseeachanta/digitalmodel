response_value_rows(envelope) -> tuple[HydrodynamicResponseValue, ...]

The ordered DOFs will be exactly surge, sway, heave, roll, pitch, yaw. The canonical time-harmonic convention will be Re(X exp(-i omega t)), with heading zero propagating along canonical +x; adapters will explicitly conjugate/rotate native results when required and record that transform. Complex ordinates will be canonical real/imaginary values; amplitude/phase will be a declared derived view. RAO and wave-excitation rows will carry frequency and heading; added-mass and radiation-damping rows will carry frequency only; restoring rows will carry DOF indices only. Inapplicable axes will be null, not sentinel values.

Wave excitation will be canonical solver-neutral response data committed by parent fields incident_wave_basis_sha256, wave_excitation_payload_sha256, wave_excitation_component_availability_sha256, and hydrodynamic_response_value_projection_contract_sha256. Long-form response kinds will be wave_excitation_total, wave_excitation_froude_krylov, and wave_excitation_diffraction, each keyed by response/envelope, DOF, frequency, and heading and carrying physical axis values, complex ordinates, and exact units. Total will be mandatory; the two decomposed components will appear together or will both be absent with a hash-bound disposition, and their complex sum will reconcile to total within a citation-backed tolerance. The parent revision will extend exact payload projection and release-set derivation, pass fresh adversarial review, and receive fresh user approval before any #1619 TDD.

For zero forward speed, the canonical incident elevation will be η(x,t)=Re(a exp(i k·(x−x_ref) − i ωt)), with intrinsic ω in rad/s, a=1 m (2 m wave height), and a positive crest at the bound reference point at t=0. Response heading will be the wave-vector propagation-toward azimuth counterclockwise from body +x on [0,2π), mapped deterministically from the inherited ENU convention. With body-outward normal n, pressure loads will use F=−∫p n dS and M=−∫(r−r_ref)×p n dS. Translational RAOs will use m/m amplitude, rotational RAOs rad/m amplitude, translational excitation N/m amplitude, and moment excitation N·m/m amplitude. Mean drift and QTFs will be excluded. Excitation/RAO/matrix payloads will bind identical execution/run, DOF, grid, basis, unit, convention, and reference commitments and pass [−ω²(M+A)−iωB+C]·RAO = excitation on exact shared coordinates without interpolation within a citation-backed tolerance after common-reference transformation.

The Capytaine adapter will not assume an identity transform. It will pin the backend version, capture primary-source/runtime convention evidence, and prove phase/heading/frame mapping with an analytic fixture. Future backend adapters will own equivalent conversion proofs. VesselMotionBoundaryEnvelope will carry only the complex RAO directly; its full hydrodynamic_response_envelope_sha256 lineage will intentionally invalidate the boundary after any excitation change.

TDD implementation sequence

  1. RED — exact envelope. Tests will assert every v3 minimum binding, commitment coverage, six-DOF order, unique row keys, finite numeric values, and parent identity/hash equality. Expected initial run: module import failure.
  2. Prerequisites. TDD will begin only after the parent excitation revision is reviewed/approved and #1618 lands its reviewed type plus canonical public-safe conformance fixture; #1619 will import those contracts rather than inventing divergent mocks.
  3. GREEN — generic immutable schema. The implementation will add typed response metadata/value rows, deterministic semantic canonicalization, and a verifier independent of Capytaine.
  4. RED — adapter conformance. A synthetic analytic BEM/RAO fixture will prove frequency/period equivalence, heading conversion, complex preservation, translational versus rotational units, matrix axes, reference point, water properties, run configuration, adapter version, and source capture.
  5. GREEN — Capytaine projection. After #1579/#1580/#1582 and the executable Capytaine environment land, a pure adapter will map explicit fields without guessing defaults or modifying the input host/spec. It will prove the evidence-derived phase, heading, and frame transform with asymmetric analytic fixtures. A second run with the same execution fingerprint will produce byte-identical normalized evidence.
  6. RED/GREEN — domain and fidelity. Tests will reject unknown phase, unit, matrix origin, restoring composition/reference, frame, reference point, host/loading/mesh mismatch, frequency/heading outside domain, flat extrapolation, heading clamp, and proxy-as-actual promotion.
  7. Tracer and independent validation. The public-safe tracer will execute or replay a pinned Capytaine result, compare against an independent public reference with declared tolerances, record unsupported features/domain, and emit either truthful benchmark/screening evidence or actual evidence only when every gate passes.

Risks and review focus

  • Blocker scope: inertia origin, restoring contract, and matrix units belong to #1579/#1580/#1582. #1619 will consume their outputs and will not duplicate them.
  • False determinism: native NetCDF ordering, floating serialization, library versions, and parallel solve order may differ. The normalized semantic projection—not native bytes—will define repeatability, while execution evidence will retain native hashes/version.
  • Numeric integrity: engineering values will not be rounded merely to force cross-machine hash equality. Finite IEEE-754 inputs will serialize through one exact shortest-round-trip representation with negative zero normalized; any numerical change will produce a new response commitment. Byte-equivalence will be required only for the same pinned execution fingerprint.
  • Validation overclaim: a semisub benchmark is not automatically the selected MODU. Fidelity and claim truth tables will remain exhaustive and fail closed.
  • Dependency on #1618: no response may be promoted without an exact reviewed host/loading envelope.
