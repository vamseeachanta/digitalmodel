Goal and authorization boundary

The program will create a public, provenance-safe, solver-neutral riser single source of truth; bind a reviewed drilling rig to a physical floating host and loading condition; execute diffraction and normalize complex RAOs; transfer vessel motion analytically to the riser top connection; generate OrcaFlex v1 models through an adapter; publish verified strength and operability results to Hugging Face; and preserve a pivot path to custom FEA or another solver.

Hard gate: this revision will not authorize implementation. The parent plan and every child plan will require adversarial review and explicit user approval. Monitoring, alarm policy, and risk-reduction evidence will remain non-blocking consumers of the immutable release.

The canonical SSOT will contain generic topology, geometry, materials, section behavior, contents, environment, loads, boundary and initial conditions, analysis intent, requested quantities, acceptance criteria, units, and coordinate frames. Solver object names, file layouts, licenses, hosts, and implicit defaults will remain outside the SSOT. Target adapters will project without back-writing.

Source handoff and three engineering envelopes

A producer-owned PublicFloaterSourceRecord will bind source URL commitment, capture date, license/redistribution, content hash, physical identity, loading/particulars/mass evidence, eligibility, ambiguity disposition, and the exact commitment consumed by the host identity.

  1. FloatingHostIdentityEnvelope will bind the source commitment, physical vessel, hull geometry, loading condition, particulars, mass properties, aliases, identity evidence, units/frame, confidence, public eligibility, ambiguity disposition, and complete-envelope commitment.
  2. HydrodynamicResponseEnvelope will bind the exact host/loading/mesh, diffraction target/adapter/projection/run evidence, ordered DOFs, frequency-heading grid, complex RAO and matrix payloads, units/conventions/frame/reference point, water properties, source, validation, usable domain, fidelity, and complete-envelope commitment.
  3. VesselMotionBoundaryEnvelope will bind the case/wave, host and response commitments, complex RAO, top-connection frame/reference, rigid-body transform and independent analytical validation, phase/time basis, coupling intent, boundary payload, coverage/domain, unsupported disposition, and complete-envelope commitment.

Seven chain fields—four IDs and three envelope hashes—will propagate unchanged through drilling request → bundle → execution request → receipt → engineering result. The release will derive exact-set commitments for all three envelope classes.

Dependency flow and routing

public riser/rig sources
  → floating host → physical vessel/hull → loading condition
  → normalized riser configuration + analysis/wave case
  → diffraction execution → complex RAO/hydrodynamic response
  → analytically validated vessel-motion boundary at top connection
  → generic analysis request → target adapter/projection → bundle
  → signed request → native solver run → raw payload → receipt
  → engineering result → immutable logical release → HF → website

Drilling will require the complete host/response/motion chain. Completion and workover will declare the chain required or will carry a reviewed not-applicable disposition. Monitoring issues #1617 and #1620–#1624 will remain outside the blocking closeout DAG.

source_registry, transformations, citations, criteria, clean_room_reviews, field_lineage, riser_components, riser_configurations, analysis_cases, execution_attestations, strength_results, operability_results, withheld_sources, floating_hosts, hydrodynamic_responses, hydrodynamic_response_values, and vessel_motion_boundaries.

TDD test list

  • Composition: base/component hash mismatch, conflicting keys, missing ownership, cyclic DAG, and exact 17-config set.
  • Host: name-only collision, conflicting IMO, different physical units, loading mismatch, missing dimensions/displacement/CoG/inertia, and private-source leakage.
  • Hydrodynamics: host/loading/mesh mismatch; unknown matrix units, origin, restoring reference, phase, frame, sign, heading, or reference point; selected-backend conformance; out-of-domain clamp/extrapolation; proven motion-domain subset; exhaustive fidelity class and proxy-as-actual claim.
  • Motion: analytical translation, rotation, combined motion, nonzero reference offset, top connection, phase/time, heading/sign, and rotational-unit fixtures.
  • Tracer: all seven fields on the required branch, discriminator/disposition on the not-applicable branch, exact handoff expansion/equality, tamper rejection, and release exact-set derivations.
  • Results/release: unsigned/mock rejection, failed oracle rejection, sparse-cell status preservation, native-file rejection, private-value rejection, and monitoring exclusion from the blocking DAG.

Acceptance criteria

  • The v3 manifest will verify every component hash and will compose without conflict, unowned nodes, cycles, silent field removal, or implicit defaults.
  • Each producer will own exactly one boundary and consumers will reference rather than redefine upstream commitments.
  • The full physical and evidence identity chain will survive unchanged through engineering result and exact-set release commitments.
  • Actual-vessel promotion will require exact host/loading, resolved diffraction blockers, explicit conventions/domain/fidelity, and independent analytical top-connection transfer validation.
  • OrcaFlex will remain the first adapter target; another solver will consume the same SSOT and generic analysis/result contracts.
  • The immutable public release will contain exactly 17 configs and no private RAO values, client identifiers, machine paths, licensed text, or native solver artifacts.
  • Every scoped child will pass its plan gate, TDD, legal/security scan, adversarial artifact review, issue comment, and completeness gate before parent closeout.
  • Monitoring and later operating-envelope population will remain non-blocking until a fresh parent revision receives owner approval.
